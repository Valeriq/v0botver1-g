# PROJECT CONTEXT — v0botver1-g (living)

## 1) Продуктовая цель (зачем проект)
Проект — система для запуска и ведения email-кампаний с управлением через Telegram, с фоновыми задачами и AI-функциями (генерация/классификация/поддержка follow-up).

**Definition of Done (для любой задачи):**
- Поведение соответствует описанному контракту (вход/выход/ошибки).
- Изменения минимальные и локальные (small batch).
- Есть способ проверить (команда/шаги) и откатить.

## 2) Текущее устройство (не обещание "навсегда")
Сейчас проект организован как набор сервисов (API, бот, воркеры, интеграции). Архитектура может эволюционировать, но изменения должны быть поэтапными и обратимо-совместимыми.

Связи высокого уровня (примерно):
- UI (Telegram) → API (HTTP)
- Worker → API/AI (HTTP)
- Gmail-service ↔ хранилище/учёт отправок
- Redis очереди для фоновых задач

## 3) Инварианты (то, что нельзя ломать)
- Публичные контракты (API/форматы сообщений/интеграции) меняются только осознанно:
  - через версионирование, адаптер или переходный период совместимости.
- Данные:
  - не "править прод" руками;
  - изменения схемы — только контролируемо (через миграции/процедуру проекта).
- Надёжность:
  - не ухудшать логирование и диагностику;
  - не убирать важные проверки ошибок.
- Безопасность:
  - секреты не логировать, не коммитить, не дублировать в коде.

## 4) Красные зоны (требуют отдельного шага и подтверждения)
Если задача затрагивает что-то из этого списка — СТОП и согласование плана:
- docker/infra/compose, сетевые порты, деплой-скрипты
- миграции/структура БД/массовые изменения данных
- OAuth/аутентификация/токены, секреты, доступы
- очереди/ретраи/семантика job'ов (risk: дубли/потери)
- массовые рефакторы, переезды папок, переименование публичных сущностей

Формат согласования:
1) что меняем и зачем
2) риск и обратная совместимость
3) план миграции/rollback
4) минимальный тест/проверка

## 5) Принцип работы "человек + агент" (самое важное)
Агент работает итерациями и маленькими порциями:
- Сначала задаёт до 5 уточняющих вопросов, если не хватает данных.
- Предлагает план 3–6 шагов, где каждый шаг — небольшой и проверяемый.
- В текущем ответе реализует только **Шаг 1** (один инкремент).
- Не делает "улучшений заодно" и не переписывает архитектуру без запроса.
- Не трогает красные зоны без явного подтверждения.

Почему: небольшие частые изменения проще ревьюить и интегрировать, и они лучше контролируются. 

## 6) Ограничение на объём изменений (token & risk budget)
По умолчанию:
- 1 задача = 1 небольшой PR/коммит
- изменения в пределах 3–7 файлов (или одного модуля)
Если требуется больше — сначала разбиение на подзадачи.

## 7) Как менять архитектуру безопасно (если понадобится)
Любое архитектурное изменение делается "через переходный слой":
- добавь новое рядом со старым,
- поддержи оба пути,
- переключи через конфиг/фичефлаг,
- удали старое отдельным PR после стабилизации.

## 8) Проверка изменений (минимальный контракт)
Каждый PR должен содержать:
- Как проверить локально (команда/шаги).
- Какие edge cases учтены.
- Что может сломаться и как откатить.

## 9) Вывод агента (формат ответа)
Агент выводит:
- список изменённых файлов
- только изменённые фрагменты или diff
- команды проверки
- риски/побочные эффекты + rollback
